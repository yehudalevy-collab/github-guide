Git vs GitHub: What's the Difference?

Git is a version control system that runs locally on your machine. It tracks every change you make to files over time, letting you travel back in history, branch off into experiments, and merge work together. It works completely offline.

GitHub is a cloud platform built on top of Git. It hosts your repositories online, adds collaboration tools (pull requests, issues, reviews), and provides automation via Actions. Think of Git as the engine and GitHub as the garage, road, and traffic system around it.

Git (Local)

Tracks history, manages branches, handles merges. Lives on your machine. No internet required.

GitHub (Remote)

Stores repos in the cloud, enables team collaboration, pull requests, CI/CD, and project management.

Repository (Repo)

A folder that Git tracks. Contains your entire project history as a series of snapshots called commits.

Commit

A saved snapshot of your project at a moment in time. Every commit has a unique SHA hash as its ID.

Core Mental Model
Working Directory → Staging Area → Local Repo → Remote (GitHub) [Your files] ──git add──▶ [Index] ──git commit──▶ [.git/] ──git push──▶ [github.com] ◀────────────── git pull ──────────────

This three-zone model is the single most important concept in Git. Before a file is committed, it must pass through the staging area (also called the index). This gives you precise control over what goes into each commit.

Setup & Configuration

Install Git

# macOS
brew install git

# Debian / Ubuntu
sudo apt install git

# Windows: download from git-scm.com or use
winget install Git.Git

# Verify
git --version

Configure Your Identity

These values are embedded in every commit you make. Set them once globally.

git config --global user.name "Your Name"
git config --global user.email "you@email.com"
git config --global core.editor "code --wait"  # use VS Code as editor
git config --global init.defaultBranch main  # name default branch "main"

# View all your config
git config --list

SSH Key Setup (Recommended)

SSH lets you push to GitHub without typing a password every time. Set it up once and forget it.

# Generate a key
ssh-keygen -t ed25519 -C "you@email.com"

# Copy it to clipboard (macOS)
pbcopy < ~/.ssh/id_ed25519.pub

# Copy it (Linux)
cat ~/.ssh/id_ed25519.pub | xclip

# Then go to GitHub → Settings → SSH Keys → New SSH Key → paste

# Test it
ssh -T git@github.com
Tip After adding your SSH key to GitHub, clone repos using the SSH URL (git@github.com:user/repo.git) instead of HTTPS. You will never be asked for a password again.

The Core Workflow

Creating or Cloning a Repo

# Start fresh in an existing folder
git init

# Clone an existing repo from GitHub
git clone git@github.com:username/repo.git

# Clone into a specific folder name
git clone git@github.com:username/repo.git my-folder

The Daily Loop

This is the sequence you will run dozens of times per day:

# 1. Check what's changed
git status

# 2. See the actual diff
git diff                  # unstaged changes
git diff --staged        # staged changes (what will be committed)

# 3. Stage files
git add index.html        # a specific file
git add src/              # a whole folder
git add .                 # everything changed
git add -p               # interactive: stage by hunk (very powerful)

# 4. Commit with a message
git commit -m "feat: add login form"

# 5. Push to GitHub
git push origin main

Reading History

git log                              # full history
git log --oneline                   # one line per commit
git log --oneline --graph --all    # visual branch graph
git log -n 5                        # last 5 commits only
git show a3b4c5d                     # inspect a specific commit
git blame src/app.js                 # see who wrote each line
Commit Message Convention Use the Conventional Commits format: type: short description. Types include feat, fix, docs, refactor, chore, test. This makes history scannable and unlocks automatic changelog generation.

Branching

Working with Branches

A branch is a lightweight, moveable pointer to a commit. Creating a branch is instant and cheap. The rule of thumb: never commit directly to main. Always work on a feature branch.

# List all branches (* = current)
git branch
git branch -a          # include remote branches

# Create and switch (modern way)
git switch -c feature/login

# Old syntax (still works)
git checkout -b feature/login

# Switch to an existing branch
git switch main

# Rename current branch
git branch -m new-name

# Delete a branch (after merging)
git branch -d feature/login

# Force delete (even if unmerged)
git branch -D feature/login

# Push new branch to GitHub
git push -u origin feature/login  # -u sets tracking
Branch model visualization: main: A ── B ── C ────────────────── F ← HEAD \ / feature/login: D ── E ──────────────
Branch Naming Conventions Use prefixes to organize: feature/ for new features, fix/ for bug fixes, hotfix/ for urgent production fixes, docs/ for documentation, release/ for release prep. Example: feature/user-auth or fix/broken-nav.

Branching

Merging & Conflict Resolution

Merging a Branch

# Merge feature/login into main
git switch main
git merge feature/login

# Prefer a merge commit even if fast-forward is possible
git merge --no-ff feature/login

# After merging, delete the branch
git branch -d feature/login
git push origin --delete feature/login

Resolving Conflicts

A conflict happens when two branches edit the same lines of the same file differently. Git marks the conflict in the file and waits for you to decide.

# What a conflict looks like inside a file:

<<<<<<< HEAD  (your current branch)
const title = "Hello World";
=======
const title = "Hello GitHub";
>>>>>>> feature/login  (incoming branch)
  1. Open the conflicted file and choose what the final code should be (delete the markers).
  2. Stage the resolved file: git add filename
  3. Complete the merge: git commit
# Abort a merge if things go wrong
git merge --abort

# Use VS Code as a visual merge tool
git config --global merge.tool vscode
git mergetool
Important Always pull the latest main before merging your branch into it. This reduces conflicts: git switch main then git pull.

Remotes & Forks

Managing Remotes

A remote is a named link to another copy of the repo, usually on GitHub. origin is the conventional name for your primary remote.

git remote -v                                     # list remotes
git remote add origin git@github.com:u/repo.git  # add one
git remote rename origin upstream                 # rename
git remote remove origin                          # remove

# Fetch remote changes without merging
git fetch origin

# Pull = fetch + merge
git pull origin main

# Pull using rebase instead of merge commit
git pull --rebase origin main

Forking Workflow

Forking is how you contribute to someone else's repo. You get your own copy on GitHub to work in, then propose changes back via a pull request.

  1. Click Fork on GitHub to copy the repo to your account.
  2. Clone your fork locally: git clone git@github.com:YOU/repo.git
  3. Add the original as upstream: git remote add upstream git@github.com:ORIGINAL/repo.git
  4. Work on a branch, push to your fork, then open a PR to the original.
  5. Sync with upstream: git fetch upstream then git merge upstream/main

Collaboration

Pull Requests

A Pull Request (PR) is a proposal to merge a branch into another. It is the heart of collaborative GitHub work: it creates a space for code review, discussion, and automated checks before code lands in main.

Opening a Good PR

  1. Push your feature branch to GitHub.
  2. GitHub will show a banner suggesting you open a PR. Click it, or go to the Pull Requests tab.
  3. Write a clear title using Conventional Commits style.
  4. In the description: explain the why, link to the issue it closes (Closes #42), add screenshots if UI-related.
  5. Request reviewers, assign labels, and connect to a project.

PR Best Practices

Keep PRs Small

Aim for under 400 lines changed. Small PRs get reviewed faster and have fewer bugs.

Draft PRs

Open as a Draft to share work-in-progress and get early feedback without signaling it is ready.

Review Suggestions

Use "Suggest changes" in review to propose exact edits reviewers can commit with one click.

Closing Keywords

Use Closes #issue, Fixes #issue, or Resolves #issue in PR description to auto-close the linked issue on merge.

Reviewing a PR via CLI

# Install GitHub CLI first: brew install gh

gh pr list                        # see open PRs
gh pr checkout 42                 # check out PR #42 locally
gh pr view 42 --web              # open in browser
gh pr review 42 --approve        # approve
gh pr merge 42 --squash          # merge with squash

Issues, Labels & Projects

Issues are GitHub's task-tracking system. Used well, they become a complete product backlog.

Issue Templates

Create a .github/ISSUE_TEMPLATE/ folder with YAML-based templates to standardize bug reports and feature requests. For example, a bug_report.yml that forces reporters to fill in reproduction steps.

Labels

Use labels to categorize issues: bug, enhancement, good first issue, help wanted, priority: high. Create custom labels that map to your workflow.

GitHub Projects (v2)

Projects is a Kanban-style board linked to issues and PRs. You can add custom fields (priority, sprint, estimate), group by assignee, filter by label, and create multiple views (board, table, roadmap timeline). It is a complete project management layer on top of your repo.

Power Tip Use milestones to group issues into versioned releases. Track progress toward v1.0 by attaching all related issues to that milestone.

.gitignore, Stash & Reset

.gitignore

Files listed in .gitignore are never tracked by Git. Critical for keeping secrets, build artifacts, and OS files out of your repo.

# .gitignore examples
node_modules/
.env
.env.local
dist/
*.log
.DS_Store
.vscode/
__pycache__/
Never commit secrets If you accidentally commit an API key or .env file, it is in your git history forever. Use git-filter-repo to purge it, then rotate the key immediately. Tools like GitHub Secret Scanning and gitleaks can help prevent this.

Git Stash

Stash lets you shelve uncommitted changes temporarily so you can switch context without committing half-done work.

git stash                      # stash everything
git stash push -m "wip: login" # stash with a label
git stash list                 # see all stashes
git stash pop                  # apply and delete latest stash
git stash apply stash@{2}      # apply specific stash, keep it
git stash drop stash@{0}       # delete a stash

Undoing Things

SituationCommand
git add . (undo staging)git restore --staged .
Discard changes in filegit restore filename
Undo last commit, keep changesgit reset --soft HEAD~1
Undo last commit, unstage changesgit reset HEAD~1
Undo last commit, discard changesgit reset --hard HEAD~1
Revert a commit (safe, adds new commit)git revert a3b4c5d
Danger Zone git reset --hard permanently discards uncommitted changes. There is no undo. Use git revert on commits that have already been pushed, since it is non-destructive.

Tags, Releases & GitHub Actions

Tags and Releases

Tags mark specific commits as significant (usually version releases). They are stable pointers that never move, unlike branches.

# Lightweight tag
git tag v1.0.0

# Annotated tag (recommended for releases)
git tag -a v1.0.0 -m "Release v1.0.0"

# Push tags to GitHub
git push origin v1.0.0
git push origin --tags     # push all tags

# Create a GitHub Release from a tag
gh release create v1.0.0 --notes "What's new in v1"

GitHub Actions (Basics)

GitHub Actions is a CI/CD platform built into GitHub. It runs automated workflows triggered by events (push, PR, schedule, etc.) using YAML files stored in .github/workflows/.

.github/workflows/ci.yml
name: CI

on:
  push:
    branches: [main]
  pull_request:

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test

This workflow runs your test suite on every push to main and on every pull request, so you always know if a PR breaks the build before merging.

Rebase, Squash & Clean History

Rebase replays your commits on top of another branch, producing a cleaner, linear history compared to a merge commit. It is the professional standard for keeping main readable.

Basic Rebase

# Bring main's latest changes into your feature branch (no merge commit)
git switch feature/login
git rebase main

# If conflicts arise, resolve them then:
git add .
git rebase --continue

# Abort if things go wrong
git rebase --abort

Interactive Rebase (History Surgery)

Interactive rebase lets you edit, reorder, combine, or delete commits before they become part of the permanent record. Invaluable for cleaning up messy WIP commits before opening a PR.

# Rewrite the last 4 commits interactively
git rebase -i HEAD~4

An editor opens showing your commits. Each line starts with a command:

CommandWhat it does
pickKeep the commit as-is
rewordKeep the commit but edit the message
editPause and let you amend this commit
squashMerge into previous commit, combine messages
fixupMerge into previous commit, discard this message
dropDelete this commit entirely

Amending the Last Commit

# Forgot to add a file, or want to fix the commit message
git add forgotten-file.js
git commit --amend

# Amend message only
git commit --amend -m "fix: corrected commit message"

# Push amended commit (requires force if already pushed)
git push --force-with-lease   # safer than --force
Golden Rule of Rebase Never rebase commits that have been pushed to a shared branch that others are working on. Rebase rewrites commit SHAs, which will cause conflicts for your teammates. Only rebase your own local or private branches.

Reflog, Hooks & Power Techniques

Git Reflog: The Safety Net

The reflog records every position HEAD has been at, even after destructive operations. It is your ultimate recovery tool. "I accidentally deleted my branch" is almost always recoverable with reflog.

# See the full movement history of HEAD
git reflog

# Output example:
# a3b4c5d HEAD@{0}: reset: moving to HEAD~2
# f1e2d3c HEAD@{1}: commit: feat: add user profile
# 9b8a7c6 HEAD@{2}: commit: fix: nav padding

# Recover to any prior state
git reset --hard HEAD@{2}

# Recover a deleted branch
git checkout -b recovered-branch f1e2d3c

Git Hooks

Hooks are scripts that run automatically at specific points in the Git workflow. They live in .git/hooks/ and can enforce rules locally.

# Example: .git/hooks/pre-commit (runs before every commit)
#!/bin/sh
npm run lint    # block commit if linting fails

# Make it executable
chmod +x .git/hooks/pre-commit

# Use Husky to share hooks across your team (hooks are gitignored by default)
npm install --save-dev husky
npx husky init

Advanced GitHub Actions

# Matrix builds: test against multiple Node versions at once
strategy:
  matrix:
    node-version: [18, 20, 22]
steps:
  - uses: actions/setup-node@v4
    with:
      node-version: ${{ matrix.node-version }}

# Caching dependencies for faster builds
- uses: actions/cache@v3
  with:
    path: ~/.npm
    key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}

# Deploy only on push to main
if: github.ref == 'refs/heads/main' && github.event_name == 'push'

# Use secrets for credentials
env:
  AWS_ACCESS_KEY: ${{ secrets.AWS_ACCESS_KEY }}

Power Techniques

# Cherry-pick: apply a single commit from another branch
git cherry-pick a3b4c5d

# Bisect: binary search to find which commit introduced a bug
git bisect start
git bisect bad                # current state is broken
git bisect good v1.2.0        # this tag was working
# Git checks out midpoint commits; you mark each as good/bad
git bisect good               # or: git bisect bad
git bisect reset              # when done

# Worktrees: check out 2 branches simultaneously in different folders
git worktree add ../hotfix hotfix/payment-bug

# Sparse checkout: clone only part of a large repo
git clone --filter=blob:none --sparse git@github.com:u/repo.git
git sparse-checkout set src/module-a

# Submodules: embed another repo inside yours
git submodule add git@github.com:u/lib.git libs/my-lib
git submodule update --init --recursive

Branch Protection Rules (GitHub Settings)

In your GitHub repo under Settings > Branches, you can enforce rules on main:

  • Require pull request reviews before merging (minimum 1 or 2 approvals).
  • Require status checks to pass (e.g. CI must be green).
  • Require branches to be up to date before merging.
  • Restrict who can push directly to the branch.
  • Require signed commits (GPG or SSH signatures).

The Unbreakable Rules

  1. Never commit directly to main. Always use a branch and PR.
  2. Never rebase a branch that others are actively working on.
  3. Never commit secrets, API keys, or passwords. Use .env and .gitignore.
  4. Pull before you push. Always sync with remote first.
  5. Write commit messages that explain the why, not just the what.
  6. Use --force-with-lease instead of --force when force-pushing.
  7. When in doubt, check git reflog. Almost everything is recoverable.
